# CareCompanion Monorepo + iOS App Design

**Date:** 2026-04-20
**Status:** Draft

---

## Overview

Convert the existing CareCompanion Next.js app into a Turborepo monorepo that houses both the web app and a new Expo iOS app, sharing TypeScript types, an API client, and utility functions across both platforms. The same PostgreSQL database and NextAuth authentication backend serve both apps.

### Migration Strategy (safe — copy → verify → delete)
1. Scaffold root `package.json`, `turbo.json`, `tsconfig.base.json`
2. Create `apps/web/` shell and copy all web app files into it
3. Update `drizzle.config.ts` paths to be relative to `apps/web/`
4. Update any CI/CD migration scripts that referenced `./drizzle` at the root
5. Run `bun run build` from root — must pass before proceeding
6. Update Vercel project "Root Directory" to `apps/web`
7. Verify Vercel preview deploy succeeds
8. Delete original root-level web files

The `@/` path alias mapping to `src/` is preserved inside `apps/web/tsconfig.json` — zero import changes needed in existing web code.

## 3. Shared Packages

### `@carecompanion/types`
Pure TypeScript, no runtime dependencies. Types are derived from the Drizzle schema using `InferSelectModel` to stay in sync automatically — do not hand-write the interfaces, instead export the inferred types:

```ts
import type { InferSelectModel } from 'drizzle-orm'
import { users, careProfiles, medications, labResults, appointments, insuranceClaims, notifications } from '../../apps/web/src/lib/db/schema'

export type User = InferSelectModel<typeof users>
export type CareProfile = InferSelectModel<typeof careProfiles>
export type Medication = InferSelectModel<typeof medications>
export type LabResult = InferSelectModel<typeof labResults>
export type Appointment = InferSelectModel<typeof appointments>
export type InsuranceClaim = InferSelectModel<typeof insuranceClaims>
export type Notification = InferSelectModel<typeof notifications>
```

This guarantees the shared types always match the actual schema columns including: `hipaaConsentAt`, `hipaaConsentVersion`, `cognitoSub`, `onboardingCompleted`, `onboardingPriorities`, `emergencyContactName`, `emergencyContactPhone`, `pharmacyPhone`, `deletedAt`, `healthkitFhirId` (new — see Section 6).

### `@carecompanion/api`
Fetch-based client, depends on `@carecompanion/types`. Base URL is injected at instantiation so swapping to a standalone API later is a single config change.

```ts
const client = createApiClient({
  baseUrl: process.env.API_BASE_URL ?? 'https://carecompanion.app',
  getToken: () => SecureStore.getItemAsync('cc-session-token'), // mobile only
  // on web: omit getToken — cookies are sent automatically
})

// Usage
await client.medications.list(careProfileId)
await client.labResults.create(data)
// Note: FHIR → shared types conversion happens BEFORE calling this:
await client.healthkit.sync({ records: convertedRecords }) // records are already Medication | LabResult | Appointment
```

Auth header strategy:
- **Mobile:** reads short-lived session token from `expo-secure-store` (see Section 4 for how it gets there), attaches as `Cookie: next-auth.session-token=...`
- **Web:** browser cookies sent automatically — no token management needed

### `@carecompanion/utils`
Three focused modules:

- **`dates.ts`** — Medical-context date formatting using `date-fns`:
  `formatRefillCountdown(date)`, `formatAppointmentDate(date)`, `daysSince(date)`

- **`fhir.ts`** — Converts HealthKit FHIR R4 JSON → shared types client-side (before any API call):
  `fhirMedicationToMedication(fhir)`, `fhirObservationToLabResult(fhir)`, `fhirEncounterToAppointment(fhir)`
  Each converter must extract and preserve the FHIR resource `id` field into `healthkitFhirId` for server-side deduplication.

- **`validation.ts`** — Zod schemas matching shared types, used for form validation on both web and mobile

## 4. Authentication

### Current state (important)
The codebase uses **NextAuth v5 with Google OAuth**, not AWS Cognito. The `cognito_sub` column stores Google OAuth `sub` values. No Cognito User Pool exists.

### Session ID Normalization (critical — must implement)
Currently `session.user.id` is set to `token.providerSub` (the Google sub string). The DB `users.id` is a UUID PK. This creates two different identity shapes. Before adding the Credentials provider, the session callback must be normalized to always return the DB UUID.

**Correct approach — store the UUID in the JWT (avoid N+1 DB queries):**

```ts
// jwt callback — runs once at sign-in, result cached in encrypted JWT
async jwt({ token, account, profile, user }) {
  if (account && (profile || user)) {
    const email = (profile as Record<string, string>)?.email ?? user?.email ?? ''
    const sub = (profile as Record<string, string>)?.sub ?? user?.id ?? ''
    // Look up DB UUID once at sign-in and cache in token
    const dbUser = await db.query.users.findFirst({ where: eq(users.email, email) })
    token.dbUserId = dbUser?.id           // UUID PK — used from here on
    token.providerSub = sub
    token.displayName = (profile as Record<string, string>)?.name ?? user?.name ?? email
    token.isDemo = false
  }
  return token
}

// session callback — reads from JWT, no DB hit
async session({ session, token }) {
  session.user.id = token.dbUserId as string  // always the UUID PK
  session.user.displayName = token.displayName as string
  session.user.isDemo = token.isDemo as boolean
  return session
}
```

All existing API routes that look up data by `session.user.id` will need to be audited to confirm they query against `users.id` (UUID), not `users.cognitoSub`. This is a required pre-migration step.

### Web additions

**Email/password Credentials provider:**
```ts
Credentials({
  credentials: { email: {}, password: {} },
  async authorize({ email, password }) {
    const user = await db.query.users.findFirst({ where: eq(users.email, email) })
    if (!user?.passwordHash) return null
    const valid = await bcrypt.compare(password, user.passwordHash)
    return valid ? { id: user.id, email: user.email, name: user.displayName } : null
  }
})
```

**Registration endpoint** (`POST /api/auth/register`):
- Accepts `{ email, password, displayName }`
- Password policy: minimum 8 characters, bcrypt with 12 rounds
- Creates user row with `passwordHash` set; for email-only users, `cognitoSub` is left null (requires making the column nullable — additive migration). Do NOT store the UUID in `cognitoSub`; that field is strictly for OAuth provider identifiers.
- Returns 409 if email already exists

**Account linking:** A user who originally signed up with Google and later wants email/password access must use a "set password" flow (not registration). A `POST /api/auth/set-password` endpoint handles this, gated by an existing session.

**DB change:** Add nullable `password_hash` column to `users` table via Drizzle migration. Google-only users have `null`.

**Known issue to audit:** Potential bug in email-only sign-in flow (non-Google) — to be investigated and fixed before wiring to mobile. Check the `authorize` callback return shape and session mapping.

### iOS Auth — Session Token Extraction

NextAuth v5 encrypts the session JWT with `AUTH_SECRET`. `ASWebAuthenticationSession` (used by `expo-auth-session`) runs in a separate browser sandbox — the session cookie it receives is inaccessible to the native app's network stack. The solution: embed the one-time code in the OAuth redirect URI itself, delivered via the custom URL scheme deep link.

**Google Sign-In flow:**
```
1. Mobile opens WebBrowser.openAuthSessionAsync(
     'https://carecompanion.app/api/auth/signin/google?mobile=true',
     'carecompanion://auth/callback'
   )

2. NextAuth completes OAuth, detects mobile=true param, generates a
   one-time code (stored in Upstash Redis, 60s TTL), then redirects to:
   carecompanion://auth/callback?code=<one-time-code>
   (instead of the normal web session redirect)

3. expo-auth-session receives the deep link, extracts the code

4. Mobile calls POST /api/auth/mobile-token/exchange { code }
   → server validates code (single-use, must not be expired)
   → returns { sessionToken } (the raw NextAuth JWT string)
   → mobile writes to expo-secure-store:
     SecureStore.setItemAsync('cc-session-token', sessionToken,
       { keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY })

5. All subsequent API calls: Cookie: next-auth.session-token={sessionToken}
```

**Email/password flow on iOS:**
```
1. Mobile form calls POST /api/auth/signin/credentials { email, password, mobile: true }
2. Server authenticates, generates one-time code, returns { code }
3. Mobile calls POST /api/auth/mobile-token/exchange { code } → session token
4. Same SecureStore write as above
```

The `app.json` custom scheme is required for step 3 to work (deep link registered with iOS):
```json
{ "scheme": "carecompanion" }
```

**Same account, both platforms:** Google sign-up on web → Google Sign-In on iOS uses the same Google sub → same `users` row (matched by `cognitoSub`).

**Same account, both platforms:** Google sign-up on web → Google Sign-In on iOS uses the same Google sub → same `users` row (matched by `cognitoSub`).

### HealthKit Sync Flow
```
App open / background refresh
  → healthkit.ts requests clinical permissions (medications, labs, conditions, appointments)
  → reads FHIR R4 records from HealthKit
  → fhir.ts (from @carecompanion/utils) converts → shared types, preserving healthkitFhirId
  → @carecompanion/api POSTs to /api/healthkit/sync { records: [...] }
  → server upserts on healthkit_fhir_id — no duplicates across refreshes
  → stored in existing RDS tables
  → web app sees data immediately (same DB, same schema)
```

## 6. New API Endpoints Required

### `POST /api/healthkit/sync`
Accepts already-converted shared types (FHIR conversion happens client-side in `fhir.ts`). A `type` discriminant field is required at runtime since TypeScript unions are erased:

```ts
Body: {
  records: Array<
    | ({ type: 'medication' } & Medication)
    | ({ type: 'labResult' } & LabResult)
    | ({ type: 'appointment' } & Appointment)
  >
}
Auth: NextAuth session (same middleware as all other protected routes)
```

Server switches on `record.type` to route each record to the correct table. Upserts on `healthkit_fhir_id` — stable unique key from the HealthKit FHIR resource `id` field. Prevents duplicates across background refreshes.

**DB change:** Add nullable `healthkit_fhir_id: text` column to `medications`, `lab_results`, and `appointments` tables with a unique index. All changes are additive — no existing data affected.

### `POST /api/auth/mobile-token`
Session → one-time code exchange (see Section 4). Requires an active session cookie.

### `POST /api/auth/mobile-token/exchange`
One-time code → session token. Code expires after 60s. Uses existing Upstash Redis.

### `POST /api/auth/register`
Email/password registration (see Section 4).

### `POST /api/auth/set-password`
Account linking for existing Google users who want to add email/password (requires session).

## 7. Security & HIPAA

HealthKit clinical records are PHI. The following must be in place before the `/api/healthkit/sync` endpoint goes to production:

- **TLS enforcement:** Vercel enforces HTTPS by default — no plaintext fallback
- **Audit logging:** Each `/api/healthkit/sync` call must be logged with: user ID, timestamp, record types synced, count. Use the existing `audit.ts` module in `src/lib/`. Audit log entries must NOT include PHI (no medication names, lab values, or diagnosis strings in log fields — only counts and record types)
- **At-rest encryption:** Verify that the RDS instance has encryption at rest enabled (AWS RDS default for new instances since 2020, but must be confirmed)
- **Vercel HIPAA BAA:** Vercel offers a HIPAA BAA on Enterprise plans. This must be in place before syncing PHI through Vercel-hosted endpoints. Confirm status before production launch
- **Token security:** The `cc-session-token` in expo-secure-store must use `SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY` accessibility level
- **One-time codes:** Mobile token exchange codes must be single-use and expire in ≤60 seconds

## 11. Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Monorepo tool | Turborepo v2 + Bun workspaces | Industry standard, Vercel-native |
| Migration strategy | Copy → verify → delete | De-risks web app move |
| Auth | Google + email/password via NextAuth v5 | Existing provider + credentials extension |
| Session normalization | Always return DB UUID from session callback | Consistent identity across providers |
| Mobile session token | One-time code exchange via custom endpoints | Only safe way to extract NextAuth JWT to native |
| Mobile framework | Expo + expo-router + prebuild | HealthKit support, TypeScript, file-based routing |
| HealthKit binding | @kingstinct/react-native-healthkit | Actively maintained, covers all clinical types needed |
| FHIR conversion | Client-side (in @carecompanion/utils) | Keeps server endpoint simple, conversion tested in shared lib |
| Deduplication | healthkit_fhir_id upsert key | Stable HealthKit FHIR resource IDs prevent duplicate inserts |
| Backend | Existing Next.js API routes | No new server, mobile calls same endpoints |
| Shared packages | types (Drizzle-derived) + api + utils | Always in sync with schema, no manual drift |
| Vercel | Root directory → apps/web | Only change needed |
| HIPAA | Audit logs + BAA confirmation required before production | PHI in transit via HealthKit sync |
